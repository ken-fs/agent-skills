# 写作哲学核心

## 四大原则

### 1. 说人话 — 你奶奶能听懂

**核心思想：**
技术再复杂，也能用日常语言解释清楚。如果你奶奶听不懂，说明你还没真正理解。

**实践方法：**

✅ **好的例子：**
```
"AI模型就像一个超级勤奋的学生，它看了几百万篇文章，
记住了哪些词经常一起出现。当你问它问题时，它就从记忆里
找出最可能的答案。"
```

❌ **坏的例子：**
```
"AI模型通过大规模预训练，学习语料库中的统计规律和语义表征，
利用Transformer架构进行多头注意力机制的特征提取..."
```

**技巧：**
- 用类比：把抽象概念比作日常事物
- 用故事：通过场景和例子说明
- 用短句：一句话说一个意思
- 问自己：我妈/我奶奶能听懂吗？

---

### 2. 有温度 — 真实的人在分享

**核心思想：**
读者能感觉到屏幕后面是一个活生生的人，不是AI或写作机器。

**实践方法：**

✅ **有温度的表达：**
```
"我第一次用Claude Code时，也被各种报错搞得头大。
后来发现，原来是我没理解它的工作方式。"

"有个有意思的地方..."
"说实话，这个功能我自己也经常用错..."
"如果你也遇到过这种情况..."
```

❌ **机械冰冷的表达：**
```
"值得注意的是，该功能在使用过程中可能会遇到错误。
经过分析发现，这是由于用户对工作原理的理解不足导致的。"
```

**技巧：**
- 使用第一人称（我、我们）和第二人称（你）
- 分享个人经验和感受
- 承认不完美和局限性
- 用口语化的连接词（"不过"、"其实"、"说实话"）
- 提出问题引发思考

---

### 3. 有洞察 — 不只"是什么"，更要"为什么"和"所以呢"

**核心思想：**
好的科普文不是知识的搬运工，而是提供别人没想到的视角和理解。

**三层深度：**

**Layer 1: 是什么（What）— 基础信息**
```
"Claude Skills是一种扩展Claude能力的机制。"
```

**Layer 2: 为什么（Why）— 背后原理**
```
"为什么需要Skills？因为再强大的AI也不可能内置所有领域知识。
Skills让Claude能够快速'上岗'，获得特定领域的专业能力。"
```

**Layer 3: 所以呢（So What）— 实际意义**
```
"这意味着什么？你可以为自己的工作流程定制专属AI助手，
而不是被迫适应通用工具的局限性。"
```

**实践方法：**
- 每个关键概念都问三遍：是什么？为什么？所以呢？
- 找到"反常识"的点（大家以为X，其实是Y）
- 揭示"隐藏规则"（为什么有人做得好，有人做不好）
- 提供"行动指南"（知道这个之后，你该怎么做）

---

### 4. 有节制 — 金句在恰当时刻闪光

**核心思想：**
金句不是撒胡椒面，而是在关键时刻画龙点睛。太多金句 = 没有金句。

**金句的时机：**

✅ **恰当的位置：**
1. **开篇第一段** - 抓住注意力
2. **核心洞察之后** - 强化记忆
3. **转折点** - 引发思考
4. **结尾** - 留有余味

**金句的特点：**
- 简短有力（一般不超过20字）
- 节奏感强（对仗、押韵、重复）
- 可以脱离上下文独立理解
- 让人想转发/划线

✅ **好的金句：**
```
"工具会过时，但思维方式不会。"
"最好的文档，是让人忘记它的存在。"
"技术的终点，是让技术消失。"
```

❌ **过度使用：**
```
每个段落都以金句结尾，导致：
- 读者疲劳
- 缺乏重点
- 显得刻意
```

**节制的艺术：**
- 根据字数档位控制金句数量（见字数档位表）
- 金句之间至少间隔2-3个段落
- 不是每个段落都需要"升华"
- 有些事实就该平铺直叙

## 写作风格的融合

### 卡夫卡的精准

**特点：**
- 每个词都有存在的必要
- 技术术语准确无误
- 逻辑推导严密

**应用：**
```
✅ "API调用失败返回503状态码"
❌ "API好像出了点问题，可能是服务器挂了吧"

✅ "训练数据截至2024年4月"
❌ "训练数据比较新"
```

### 余华的朴素

**特点：**
- 短句为主，长句为辅
- 去掉华丽的修辞
- 用最简单的词说最深刻的事

**应用：**
```
✅ "代码写完了。测试通过了。但我知道还有bug。"
❌ "在经过一系列复杂的开发和严格的测试之后，虽然初步验证
    显示功能基本符合预期，但基于丰富的工程经验，我深知..."
```

### 马尔克斯的神奇

**特点：**
- 在平凡中发现魔幻
- 意外的比喻
- 让技术充满故事性

**应用：**
```
✅ "GitHub Copilot第一次自动补全我的代码时，我愣了三秒。
    就像看着一个陌生人准确地说出了我接下来要说的话。"

✅ "训练一个大模型，就像养育一个天才儿童。你喂给它整个
    互联网的知识，但它学到什么、会说什么，你永远无法
    完全预测。"
```

## 实战检查清单

完成草稿后，逐项检查：

### 说人话检查
- [ ] 有没有不必要的术语？
- [ ] 每个专业词汇都解释了吗？
- [ ] 让非专业人士读一段，他们能懂吗？

### 温度检查
- [ ] 有没有第一人称和第二人称？
- [ ] 有没有个人经验和感受？
- [ ] 读起来像人说话，还是像AI生成？

### 洞察检查
- [ ] 不只说"是什么"，还说了"为什么"和"所以呢"吗？
- [ ] 有没有提供新的视角或理解？
- [ ] 读者读完能获得行动指南吗？

### 节制检查
- [ ] 金句数量符合字数档位吗？
- [ ] 金句之间间隔合理吗？
- [ ] 有没有为了金句而金句？

## 字数档位与金句配比

| 档位 | 字数 | 金句 | Section | 金句密度 |
|------|------|------|---------|----------|
| A. 精炼版 | 1200-2000 | 2-3句 | 2-3个 | 约600-800字/句 |
| B. 标准版 | 2000-4000 | 3-5句 | 3-4个 | 约500-800字/句 |
| C. 深度版 | 4000-7000 | 5-7句 | 4-6个 | 约600-1000字/句 |
| D. 长文版 | 7000-10000 | 6-8句 | 5-8个 | 约900-1400字/句 |

**原则：**
- 篇幅越长，金句密度越低（避免疲劳）
- 每个主要Section至少1句金句
- 开篇和结尾必有金句
